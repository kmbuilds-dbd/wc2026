4 · Deviations from the spec

4.1 Scoring rebalance

README does not specify exact point values — the spec is silent on the scoring tuning. Initial implementation went with what felt natural; after a Monte Carlo exercise (50k simulated tournaments) we found the original distribution gave a lucky tournament-winner picker a near-equivalent score to a fully-skilled player who happened to pick the wrong champion (621 vs 607 mean). Skilled+lucky won 99.9% of head-to-heads — i.e. tournament-winner identification was effectively decisive.

Three tunings were applied in commit 893f888:

RuleBeforeAfterRationale
BRACKET_SF / BRACKET_FINAL80 / 16060 / 80Flattens the winner-chain bonus from 310 → 210.
TOURNAMENT_WINNER10050Halves the binary-winner bonus.
TOURNAMENT_FINALIST (new)—50Picking the losing finalist now scores partial credit — a smoother gradient.
GROUP_PICK_EXACT58Raises the floor on skill-based categories.
LINEUP_GOAL / LINEUP_ASSIST10 / 515 / 8Same — bigger reward for active prediction.

Result after rebalance (50k trials): skilled-unlucky now beats lucky-but-bad by ~270 pts on average (was −14 before). Skilled+lucky win share dropped from 99.9% → 90.8%. This is a divergence, not a fix — the original values could have been left alone. The user reviewed the simulation and chose to apply the rebalance.

4.2 Bracket lock semantics

README's locks.ts bullet says "every write path must call isLocked() before mutating" and implies a single lock point per category. The actual implementation has a three-state model (pending / open / locked) for bracket and KO lineups because the binary lock-or-not couldn't express "matchups not yet decided." isLocked("bracket") still returns a boolean for write-protection (true if not in the open window); UI uses getBracketWindow() / getLineupWindow() for the tri-state rendering.

4.3 Round enum expanded

README mentions only R32–Final for lineups. Schema now also accepts "group". Drizzle confirmed this is a TypeScript-level enum only — SQLite stores plain TEXT — so no migration was needed. The added round flows through LineupRound type, computeAliveTeams, LINEUP_ROUNDS, the lineup picks page, and the locked roster view.

5 · Design decisions made during implementation

5.1 Data provider deferral

README assumes api-sports.io Free tier supplies match data, lineups, top scorers (100 req/day). In practice the Free tier blocked all 2026+ seasons with "Free plans do not have access to this season, try from 2022 to 2024." Five other providers (balldontlie GOAT, TheSportsDB, bzzoiro, Apify macheta) were evaluated; none of the free options actually cover WC 2026.

The two realistic paths:

  • api-football Pro $19/mo — covers everything, our existing client code already targets it. Subscribe ~Jun 9, renew once, cancel after the tournament.
  • $0 hybrid — hardcoded fixtures from FIFA's published schedule + admin form for entering match results manually via /api/admin/match-result. ~4h rewrite, ~2h admin time during tournament.

Decision held until ~Jun 7–8 per progress.md. The choice doesn't block any of the pick UI or scoring work that's been built — those work fine with the snapshot data already in place.

5.2 Empty-state design

Several pages (/odds, /matches, /leaderboard) can render before any data exists. Each has a dedicated empty state explaining what's missing and how to populate it (e.g. "POST /api/cron/refresh-odds with the cron secret"). Chosen over hiding the page entirely so admin testing is possible pre-launch.

5.3 Top Scorer / Golden Glove store player IDs

Schema stores topScorerPlayerId and goldenGlovePlayerId as integers (not team IDs). For Golden Glove this requires a player→team join at scoring time because the award is team-based (most clean sheets); we credit any GK of the leading team. Decision: keep the player-id contract for UI uniformity even though the actual award is team-level.

5.4 The Odds API key discovery

The initial implementation assumed the base sport soccer_fifa_world_cup with markets=outrights. The Odds API rejected this with 422 INVALID_MARKET_COMBO; outrights are exposed as their own sport keys (soccer_fifa_world_cup_winner, …). Refactored to a hardcoded sport-key → market map, with the diagnostic probe endpoint kept around to discover new keys as the tournament approaches.

6 · Tradeoffs

6.1 Pending UI: coming-soon panel vs disabled inputs

When bracket / KO lineups are in the pending state, two render options were considered: (a) the 31 (or 4) dropdowns disabled with a banner, or (b) a single coming-soon panel. Picked (b) because the dropdowns would have shown team names that the user can't meaningfully assign yet (matchups unknown), creating visual noise. Tradeoff: users lose the ability to preview the bracket shape until it opens.

6.2 Bracket opens-at as a constant vs derived from match data

Could have set bracket-opens-at to "moment the last group-stage match finishes" by polling match status. Chose a hardcoded Jun 28 00:00 UTC instead because (i) no data provider is wired, (ii) it's predictable, (iii) the FIFA schedule is fixed and won't move. Tradeoff: if a group-stage match runs into extra time and finishes after midnight UTC, the window technically opens before all matchups are mathematically resolved. Acceptable for an amateur-pool tournament.

6.3 Scoring legend: live values vs documented constants

The ScoringLegend component imports RULES directly so the values shown to users can never drift from what's actually scored. Tradeoff: every scoring rule rebalance is a one-line edit in rules.ts and the UI updates automatically — but it also means any unintended change to rules.ts silently changes the public-facing legend.

6.4 /matches page: rollups + per-match vs split into two pages

Considered splitting into /matches (per-match drilldown) and /leaderboards (top scorer / clean sheet). Combined onto one page because (i) the user explicitly asked for "all relevant stats... list them out" on a single surface, (ii) rollups are cheap to compute alongside the match list, (iii) it reduces nav surface area.
